## 📝 설명


### 1. HTTP

> html, css, js, png, jpes와 같은 파일이나 문서들이 링크를 통해 서로 연결되어 있는 텍스트 등 일반적인 텍스트를 뛰어넘는 구조화된 텍스트를 전송하기 위해 사용되는 통신 규약

> - **사용 이유**: 웹 서버와 클라이언트가 하이퍼텍스트 문서를 주고받을 수 있도록 정의된 규칙에 따라 문서를 전송하기 위함
> - **사용 방법**: 클라이언트가 'URL'을 통해 특정 웹 주소로 요청을 보냄

```text
클라이언트 (요청) ←————————→ 서버 (응답)
```

### 1-1. HTTP Message

> 클라이언트와 서버 간에 파일 등을 주고 받는 통신의 기본 단위

> - **사용 이유**: 클라이언트와 서버 간 데이터를 교환하기 위해

 **구조**

  | 순서 | 이름 | 내용 |
  |---|---|---|
  | 1 | Start Line | 요청이나 응답의 상태를 나타내는 첫 번째 줄 |
  | 2 | HTTP headers | 메시지 바디를 요약하는 헤더들의 집합 |
  | 3 | Empty Line | 헤더와 본문을 구분하기 위해 존재하는 빈 줄<br>이전까지의 모든 헤더가 전송되었음을 알리고 본문(Body) 부분이 시작된다는 것을 의미 |
  | 4 | Body | 데이터나 문서 등 실제 내용이 포함됨.<br>HTML 문서, JSON 데이터 등이 본문에 담길 수 있음 |

| 헤더 유형 | 설명 |
|---|---|
| Request Headers (요청 헤더) | 클라이언트가 서버에게 요청을 보낼 때 함께 전송되는 헤더<br>요청의 성격과 목적을 설명하고 부가적인 정보를 제공함 |
| Response Headers (응답 헤더) | 서버가 클라이언트에게 응답을 보낼 때 포함하는 헤더<br>상태 코드, 서버 정보, 클라이언트가 어떻게 응답을 처리해야 할 지에 대한 정보 제공 |
| General Headers (일반적인 헤더) | 요청과 응답 모두에서 사용되는 일반적인 헤더<br>메시지 전체에 대한 정보를 제공하며, 캐시, 연결 유지 등과 관련된 설정 포함 |
| Representation Headers (대표 헤더) | 요청과 응답 메시지의 본문 데이터(representation)와 관련된 헤더<br>데이터의 형식, 언어, 압축 등을 나타내어 클라이언트가 데이터를 올바르게 해석하고 처리할 수 있도록 도와줌 |

- **대표 헤더가 필요한 이유**
  - 본문 데이터와 관련된 헤더들은 서버가 요청을 처리하기 전 필요한 정보를 제공하는 역할
  - 서버가 요청을 처리하기 전 단계에서 이러한 헤더 정보는 미들웨어에서 전처리될 수 있음

| 헤더 이름 | 설명 |
|---|---|
| Host | 요청하려는 서버 호스트 이름과 포트 번호 |
| User-Agent | 클라이언트 프로그램 정보 (브라우저 등) |
| Accept | 클라이언트가 처리 가능한 미디어 타입 종류 |
| Accept-Language | 클라이언트가 처리 가능한 언어 종류 |
| Accept-Encoding | 클라이언트가 서버로부터 응답을 받을 때 지원하는 데이터 압축 방식 |
| Connection | 클라이언트와 서버 간의 TCP 연결을 관리하는 방법을 지정 |
| Content-Type | 본문(body)의 데이터 유형을 지정 |
| Content-Length | 본문 길이를 바이트 단위로 나타냄 |
| Upgrade-Insecure-Requests | 브라우저에게 현재 보안되지 않은 HTTP 연결을 보안된 HTTPS 연결로 업그레이드하도록 요청하는 지시 |

### 1-2. HTTP Request Method

> 웹 서버에게 어떤 작업을 수행하길 원하는 지 알리는 방법을 정의하는 명령 규칙
> URL과 함께 GET, POST, PUT 등의 메소드를 명시해 서버에게 특정 작업을 요청함

> - **사용 이유**: 자료 요청, 제출, 삭제 등 구체적인 작업을 명확하게 지시하기 위해

| Method | 사용 이유 | 사용 방법 | Request Body |
|---|---|---|---|
| GET | 데이터 조회 요청 | - 요청 파라미터를 Query String의 형태로 URL에 포함시켜 전송<br>- 해당 URL에 대한 처리를 수행하고, 클라이언트에게 요청한 데이터를 응답으로 보냄 | ❌ |
| POST | 리소스 생성 및 업데이트 | - Request Body의 데이터를 확인하고, 해당 데이터에 따른 처리를 수행한 후 응답을 클라이언트에게 보냄 | ✅ |
| PUT | 리소스 전체 업데이트 | - Request Body의 데이터를 파싱하고, 해당 데이터를 기반으로 리소스 업데이트 | ✅ |
| PATCH | 리소스 부분적인 수정 및 업데이트 | - Request Body의 데이터를 파싱하고, 해당 리소스의 일부분을 수정 또는 업데이트 | ✅ |
| DELETE | 리소스 삭제 | - URL에 삭제하고자 하는 리소스의 식별자를 포함해 전송<br>- 해당 리소스를 식별하고 삭제 | 🔺 |

### 1-3. HTTP Status Code

> 웹 통신 규약에서 요청의 처리 상태를 나타내는 코드

> - **사용 이유**: 클라이언트가 웹 서버에 요청을 했을 때 통신 결과를 웹 서버가 클라이언트에게 명확하게 전달하기 위해서
>   - 클라이언트는 서버로부터 받은 상태 코드를 확인해 요청 처리 상태를 확인하고 상황에 맞게 처리 가능
>   - 상태 코드를 통해 사용자에게 적절한 안내나 오류 처리를 제공 가능

> - **사용 방법**
- 요청 처리 결과를 나타내는 HTTP Status Code를 응답 메시지의 헤더에 포함시켜 클라이언트에게 전송
  | 코드 | 설명 |
  |---|---|
  | ```1xx``` | 정보 메시지 (Informational) |
  | ```2xx``` | 성공 (Successful) |
  | ```3xx``` | 리다이렉션 (Redirection) |
  | ```4xx``` | 클라이언트 오류 (Client Error) |
  | ```5xx``` | 서버 오류 (Server Error) |
  
- 자주 사용하는 상태 코드
  | Status Code | 설명 | 예시 |
  |---|---|---|
  | ```200``` | 요청이 성공적으로 처리됨 |  |
  | ```201``` | 요청이 성공적으로 처리 & 그 결과 새로운 리소스 생성 | - 게시글 추가 |
  | ```202``` | 요청을 수신했으나 그에 대한 처리 불가<br>(다른 프로세스에서 처리 중이거나 서버가 요청을 처리하고 있는 경우) | - 게시글 추가: 많은 데이터를 한 번에 보내 서버에서 처리하는 데 시간이 걸리는 경우 |
  | ```204``` | 헤더는 유효하지만 요청에 대해 보내줄 수 있는 내용 없음 | - 게시글 업데이트 |
  | ```400``` | 잘못된 문법으로 인해 서버가 요청을 이해 불가 | - 게시글 작성: 제목이나 본문 데이터가 클라이언트에서 서버로 전달이 안된 경우 |
  | ```401``` | 인증되지 않은 접근 | - 게시글 조회: 로그인 하지 않고 회원만 접근 가능한 글을 조회할 때 |
  | ```403``` | 인가되지 않은 접근 | - 관리자 페이지에 일반 유저가 접근 시도할 때 |
  | ```404``` | 요청받은 리소스를 찾을 수 없음 | - 삭제된 게시글 조회 |
  | ```429``` | 지정된 시간 안에 너무 많은 요청을 보냄 | - 사용자가 잘못된 비밀번호로 많은 요청하는 경우 |
  | ```500``` | 서버가 처리 방법을 알 수 없음 |  |
  | ```501``` | 요청 방법이 서버에서 지원되지 않아 처리 불가 | - 지원하지 않는 HTTP 메소드로 요청이 온 경우 |
  | ```503``` | 서버가 요청을 처리할 준비가 되지 않았음 (일시적 사용 불가능) | - 서버에 많은 트래픽이 몰리거나 서버 유지보수 상황인 경우 |


### 1-4.URL

> 리소스의 위치와 종류를 나타내는 주소

```text
http://www.example.kr:80/path/to/example.html?key1=value
|             |            |                      |
scheme      domain     path to resource        parameters
```

| 구성 요소 | 설명 | 예시 |
|---|---|---|
| scheme | 브라우저가 리소스를 요청할 때 사용해야 하는 프로토콜 | ```http```, ```https``` | 
| domain | 요청하는 웹 서버 | ```www.example.kr``` | 
| port | 웹 서버 리소스에 접근 시 사용되는 게이트 | HTTP: ```:80```<br>HTTPS:```:443``` | 
| 리소스 경로 | 요청하는 리소스 위치 | ```/path/to/example.html``` | 
| parameters | 추가 파라미터 | ```?key1=value``` |

### 2. REST API

> REST 아키텍처 스타일을 따르는 응용  프로그램 인터페이스

> - **사용 이유**: 개발자 간의 일관된 통신 규칙을 마련해 업무 효율을 높이고, 단순한 URL 구조를 통해 요청의 목적을 쉽게 이해할 수 있게 하기 위함

| 항목 | 설명 |
| --- | --- |
| 일관된 통신 규칙 제공 | - HTTP 표준을 기반으로 통신 규칙을 제공<br>- 개발자 간의 협업을 원활하고, 코드의 가독성/유지보수 용이하게 함 |
| 단순한 URL 구조 | - 자원을 명확히 표현하는 단순한 URL 구조<br>- 요청의 목적을 쉽게 파악 가능<br>- API 사용과 이해 직관적 |
| 유연성과 확장성 | - 클라이언트와 서버가 독립적으로 작동함<br>- 서로 영향을 주지 않고 개별적으로 기능을 개발/변경 가능 → 시스템 설계 유연 |
| HTTP 메서드 활용 | - GET, POST, PUT, DELETE 등 HTTP 메서드를 사용해 작업의 의도 명확히 표현 → 데이터 처리 과정 명확 |
| 다양한 플랫폼 지원 | - 플랫폼에 종속 X<br>- 사용 범위가 넓고 접근성 뛰어남 |

- **REST**
  - 자원을 이름으로 식별하고, 해당 자원의 상태(정보)를 구조화된 형식으로 표현하여, 서버와 클라이언트 간에 정보를 교환하는 방식
- **API**
  - 응용 프로그램 인터페이스
  - 운영체제와 응용 프로그램 사이의 통신에 사용되는 언어나 메시지 형식
- **소프트웨어 아키텍처 스타일**
  - 시스템을 설계할 때 사용하는 구조적 접근 방식
  - 특정한 설계 원칙과 규칙을 따름
  - 데이터 흐름, 구성 요소 간의 상호작용, 확장성 등을 효과적으로 관리하기 위한 가이드라인 제공
- **REST 아키텍처 스타일**
  | 항목 | 설명 |
  | --- | --- |
  | Uniform Interface | - 모든 API가 일관된 방식으로 통신하도록 해서 사용하는 데 있어 단순성, 상호 운용성을 보장해야 함 |
  | Client-Server | - 클라이언트와 서버 독립적으로 구분, 각각의 개발 확장 편리, 사용자 인터페이스와 데이터 저장소 분리되어 있어야 함 |
  | Stateless | - 서버는 클라이언트의 상태 정보를 저장하지 않아야 함<br>- 각각의 요청이 독립적이며 필요한 모든 정보를 포함해야 함 |
  | Cacheable | - 응답은 캐싱이 가능하도록 명시되어 있어야 함 |
  | Layered System | - 클라이언트는 서버가 단일 서버인지 다중 계층을 통해 데이터를 전달하는 중계서버를 거치는지 알 수 없어야 함 |
  | Self-Descriptiveness | - 메시지 자체가 충분한 정보를 담고 있어, 클라이언트가 어떻게 처리해야 할 지를 알 수 있어야 함 |

**사용 방법**
1. URI는 정보의 자원 표현: 리소스명은 동사가 아닌 명사

    | 항목 | 예시 |
    | --- | --- |
    | 적절한 예시 | ```/users```, ```/users{userId}/posts``` |
    | 부적절한 예시 | ```/getUsers``` |

2. 자원의 행위는 HTTP 메소드 이용: GET, POST, DELETE, PUT, PATCH로

    | 항목 | 예시 |

### 3. JSON

> JavaScript Object Notation
> 자바스크립트 객체 문법으로 구조화된 데이터를 표현하기 위한 문자 기반의 표준 포맷

> - **사용 이유**: 경량화, 구조화된 데이터 형식으로 데이터를 주고 받기 위해

  | 특징 | 설명 |
  | --- | --- |
  | 언어 독립성 | 다양한 프로그래밍 언어로 작성된 시스템 간에 데이터를 쉽게 주고받을 수 있게 해줌 |
  | 읽고 작성 쉬움 | 사람이 읽고 쓰기 쉬운 텍스트 형식으로 되어 있어, 데이터 구조를 쉽게 이해 가능 |
  | 손쉬운 데이터 교환 | 네트워크를 통해 데이터 교환 시 좋으며, 데이터를 쉽게 파싱하고 사용 가능 |

```json
  [
  	{
  		"id": 1,
  		"name": "TIL"
  	},
  	{
  		"id": 2,
  		"name": "Weekly Challenge"
  	}
  ]
```
- 사용
    - Key-Value: 각 데이터에 접근하고 식별하는 역할
    - 배열 [] 사용 가능
- 특징
    - 데이터를 명확하고 체계적으로 표현 가능
    - 텍스트 형식이기 때문에 파일로 저장하거나 네트워크를 통해 쉽게 전송 가능
