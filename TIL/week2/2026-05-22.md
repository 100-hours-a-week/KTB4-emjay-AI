### 1. FastAPI

> Python으로 빠르고 쉽게 웹 API 서버를 만들 수 있는 웹 프레임워크 -> '빠르게 API 서버를 만들 수 있도록'

> FastAPI: 애플리케이션을 실행하는 서버 환경 -> Uvicorn과 같은 ASGI 서버 위에서 동작함<br>
> FastAPI 자체는 설계도만 제공하고 실행 엔진 없음 -> 실제로 외부 요청을 받아서 앱을 돌려주는 역할은 ASGI 서버가 수행


| 단어 | 설명 |
|---|---|
| API | - Application Programming Interface<br>- 서로 다른 소프트웨어 애플리케이션들이 정해진 규칙에 따라 서로 소통하고 정보를 교환할 수 있도록 돕는 통신 규약<br>- 소프트웨어 내부 기능(데이터 요청/기능 실행 등)을 외부에서 안전하게 사용할 수 있도록 |
| 프레임워크 | - 소프트웨어 개발에서 기본 뼈대와 실행 흐름을 미리 정해둔 개발 툴<br>- 애플리케이션의 아키텍처와 실행 순서를 미리 정해둠<br>- 프레임워크를 사용해서 개발할 경우 공통적으로 들어가는 기능을 제공하여 개발자는 핵심 로직만 구현하면 됨 |


| 단어 | 처리 | 프레임워크 | 설명 | 특징 |
|---|---|---|---|---|
| ASGI | ```비동기``` | FastAPI | - Asycnronous Server Gateway Interface<br>- 파이썬 웹 서버, 프레임워크, 애플리케이션 간의 비동기 표준 인터페이스 | DB 조회, 외부 API 요청 등 시간 오래 걸리는 작업을 기다리는 동안 다른 요청 처리 가능 |
| WSGI | ```동기``` | Flask | - Web Server Gateway Interface<br>- 파이썬 웹 서버, 프레임워크, 애플리케이션 간의 동기 표준 인터페이스 | 한 요청을 처리하는 동안 다른 요청은 대기 |

> - **사용 이유**: 비동기 기반 고성능 처리와 타입 기반 검증/자동 문서화를 통해 불필요한 코드를 줄이고 API 서버를 쉽게 개발/관리할 수 있기 때문
>   - FastAPI에서는 Uvicorn과 Starlette 기반으로 대규모 동시 요청 처리 가능
>   - Python 타입 힌트를 활용해 요청/응답을 자동으로 검증하고 API 문서를 자동으로 생성해줌
>   - `async`/`await` 문법으로 고성능 비동기 처리 가능 -> 실시간 서비스 구현에 용이  

- **Uvicorn**
  - Python에서 사용되는 웹 애플리케이션 서버(WAS, Web Application Server)
  - FastAPI 같은 ASGI 기반 웹 애플리케이션을 실행하는 역할
  - FastAPI 같은 비동기 웹 프레임워크가 만든 앱을 실제로 구동해 외부 요청을 받아 처리함 => '개발자가 작성한 코드를 웹에서 동작하도록 연결해주는 실행 엔진 역할'
    - FastAPI: 웹 로직 담당 프레임 워크 <-> Uvicorn: 그 애플리케이션을 실제 구동시키는 서버

- **Starlette**
  - Python용 ASGI 프레임워크
  - 웹 애플리케이션을 직접 만들기 위한 최소한의 핵심 기능(라우팅, 미들웨어, 요청/응답 객체, 세션, WebSocket 지원 등)을 제공함
  - FastAPI는 Starlette 위에서 동작하며, 실제 요청/응답 처리, 라우팅, 미들웨어 같은 핵심 기능은 Starlette이 담당함
  - 'FastAPI는 그 위에 Pydantic 기반 데이터 검증, 타입 힌트 지원, 자동 문서화 등 고급 기능을 얹은 것'

```text
  사용자 코드 ------> FastAPI ------> Starlette ------> ASGI Server (Uvicorn)
                              |                  |                      |
             FastAPI는 Starlette 사용        Starlette는 ASGI 표준 사용    실제 실행은 Uvicorn 같은 서버 담당
```

- **API 문서**
  - API를 어떻게 사용해야 하는지 알려주는 설명서
  - 엔드포인트, 요청 방법, 파라미터, 응답 형식, 에러 코드 등을 정리한 문서
  - 개발자가 API를 쉽게 이해하고 활용할 수 있도록 가이드 제공


- **async/await**: Python에서 비동기 프로그래밍을 쉽고 효과적으로 작성할 수 있게 해주는 키워드

  | 단어 | 설명 |
  |---|---|
  | `async` | - 함수를 비동기 함수(코루틴)로 정의<br>- `async`로 정의된 함수는 호출 시 코루틴 객체를 반환하며, 실제 실행은 `await`을 만나서 일시 중단/재개하는 방식으로 수행 |
  | `await` | - 비동기 함수 내부에서 사용<br>- 특정 작업이 완료될 때까지 기다렸다가 다음 코드로 진행<br>- `await`을 만나면 현재 코루틴은 일시 중단되고, 이벤트 루프에 제어권을 넘겨 다른 작업을 처리할 수 있게 함 |

### 2. FastAPI에서 Pydantic 모델로 입력 데이터 검증하기

> FastAPI에서는 Pydantic Model을 사용해 Request Body를 검증함

> **Pydantic Model**
> - Request Body의 데이터 구조와 타입을 정의하는 DTO(Data Transfer Object)의 역할을 함
> - 요청으로 전달된 데이터를 해당 모델로 변환하는 과정에서 자동으로 유효성 검사 수행 => '개발자가 if문이나 별도의 검증 로직을 작성할 필요 X'

- **사용 방법**
```python
from pydantic import BaseModel

class PostCreate(BaseModel):
  title: str
  content: str

@router.post("/posts")
def create_post(payload: PostCreate):
  ...
```
  - Request Body의 구조와 규칙을 모델로 선언 -> FastAPI는 이를 기반으로 *라우터 함수가 실행되기 전에 자동으로 검증 수행*
    - Request Body 존재 여부
    - 필수 필드(title, content) 누락 여부
    - 각 필드의 타입 일치 여부
  - if) 검증 실패 시 => 라우터 함수는 실행 X & FastAPI는 자동으로 `422 Unprocessable Entity` 응답 반환

| DTO | 설명 |
|---|---|
| 정의 | - Data Transfer Object<br>- 계층 간에 데이터를 전달하기 위해 사용하는 객체 |
| 목적 | 주로 API 요청(Request)이나 응답(Response)에서 데이터의 구조를 명확히 정의하는 목적 |
| 사용 | - FastAPI에서는 Pydantic Model이 DTO 역할<br>- Request Body를 Pydantic Model로 선언 -> 해당 모델이 요청 데이터의 구조, 타입, 필수 여부를 정의하는 DTO가 됨 |

### 2-1. 입력 데이터 검증하기

```python
from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI()


class PredictRequest(BaseModel):
    text: str = Field(
        ...,
        min_length=1,
        max_length=500,
        description="감정 분석할 문장"
    )


@app.post("/predict")
def predict(request: PredictRequest):
    return {
        "input_text": request.text,
        "result": "positive"
    }
```

| 코드 | 의미 |
|---|---|
| BaseModel | Pydantic 모델을 만들기 위한 기본 클래스 |
| text: str | text 값은 문자열이어야 함 |
| Field(...) | 필수 입력값 |
| min_length=1 | 빈 문자열 허용 X |
| max_length=500 | 500자 초과 허용 X |

즉, 아래와 같은 요청만 정상적으로 처리함
```json
{
  "text": "오늘 기분이 좋아요"
}
```

| 잘못된 요청 | 예시 |
|---|---|
| 필수값 누락 | {} |
| 빈 문자열 | { "text": "" } |
| 너무 긴 문자열 | { "text" : "500자를 초과하는 아주 긴 문장 ..." } |


### 2-2. Optional[객체] = Body(None)의 문제

`request: Optional[RequestModel] = Body(None) 패턴의 문제`
  - FastAPI가 원래 자동으로 수행해 주는 요청 Body 검증 흐름 자체를 끊어버림

#### 1) 정상

```python
def signup(payload: SignUpRequest):
    ...
```
위와 같은 라우터 코드가 있을 때,
요청이 들어올 경우 다음의 과정을 거침

```text
1. HTTP 요청의 Body를 읽음
2. Body를 SignUpRequest라는 Pydantic Model로 변환하려고 시도
3. 변환 과정에서 Pydantic이 자동으로 검증 수행
   - Body가 존재하는지
   - 필수 필드가 누락되지 않았는지
   - 각 필두의 타입이 맞는지
5. 검증에 실패하면, 라우터 함수는 호출 X
6. 검증에 성공한 경우에만, payload 인자에 '검증이 끝난 객체' 주입
```

> 즉, 위의 상황에서는 payload 인자에 들어오는 값이 항상 다음을 보장함
>  - None이 아님
>  - 구조 명확
>  - 타입이 이미 검증됨


#### 2) 문제 상황

```python
def signup(signup_data: Optional[SignUpRequest] = Body(None)):
    ...
```
이 선언은 FastAPI가 전혀 다른 의미로 해석함
  - 이 Body는 필수가 아님: Body 없어도 괜찮음
  - 검증에 실패해도 함수는 호출될 수 있음

```text
실제 싫행 시 signup_data 인자에 다음과 같은 값들이 들어올 수 있음
  - 요청 Body가 없을 경우: signup_data = None
  - 요청 Body가 있지만 구조가 불완전한 경우: signup_data = None 이거나 예상치 못한 형태
```

> 즉, signup_data 인자는 '있을 수도 있고, 없을 수도 있는 값'이 됨
>  - FastAPI가 자동 검증을 책임지지 X

#### 3) 정리

- **정상 FastAPI 구조**
  ```text
  라우터 함수 실행 전
  
  요청 Body -> Pydantic 변환 -> 자동 검증 -> 검증된 값 주입
  ```
  - 자동 검증, 자동 에러 처리, 자동 문서화 가능
- **Optional[RequestModel] = Body(None)**
  - 자동 검증 흐름 끊기고, 검증 책임이 모두 라우터 코드로 넘어옴
  - FastAPI를 쓰면서도 Flask처럼 모든 검증/응답을 수동으로 처리하는 코드 구조를 만들게 됨 => 프레임워크가 제공하는 핵심 장점 포기

### 3. AI 모델 서빙 시 발생할 수 있는 잘못된 요청 방지

- AI 모델 서빙 시 입력 데이터 검증이 중요한 이유
  ```text
  AI 모델 서빙 API 동작 흐름
  
  클라이언트 요청 -> FastAPI -> 입력값 검증 -> 전처리 -> AI 모델 추론 -> 결과 반환
  ```
  - Pydantic 검증이 없다면, 잘못된 요청이 그대로 모델에 들어갈 수 있음
    ```json
    {
      "text": null
    }
    ```
    - 위 요청이 그대로 모델 전처리 코드에 들어간다면?
      - `tokens = request["text"].split()`
        - `None`에는 `.split()`이 없기 때문에 서버 오류 발생
        - `AttributeError: 'NoneType' object has no attribute 'split'`
    - 하지만, Pydantic을 사용한다면?
      - `text`가 문자열인지 먼저 검사 -> AI 모델까지 잘못된 값이 전달되지 X
- 잘못된 요청 방지
  - Pydantic 검증: 잘못된 요청을 '사전에 차단'
  - Exception Handler: 차단된 오류를 '일관된 방식으로 처리'

### 3-1. Pydantic 기반 입력 검증

#### 1) Pydantic으로 입력 타입 검증
- Pydantic 사용 -> 이러한 요청은 AI 모델까지 전달되지 않고, FastAPI가 먼저 오류를 반환함
- 예시)
  - 필수값 누락
  - 타입 오류
  - 빈 문자열
  - 너무 긴 입력

#### 2) 이미지 URL 검증
- 이미지 모델일 경우 -> URL 형식 검증

```python
from pydantic import BaseModel, HttpUrl

class ImageRequest(BaseModel):
    image_url: HttpUrl
```

- 잘못된 요청:
  ```json
    {
      "image_url": "not-url"
    }
  ```
  - 이 경우 URL 형식이 아니므로 모델 실행 전 차단

#### 3) 선택값 제한
- 모델 종류 제한
  - 예시) `model_type: Literal["sentiment", "summary", "tranlate"]`
  - model_type에 정해진 값만 들어올 수 있음

#### 4) 직접 검증 로직 추가
- 직접 검증
  ```python
    from fastapi import HTTPException

    @app.post("/predict")
    def predict(request: PredictRequest):
        if request.text.strip() == "":
            raise HTTPException(
                status_code=400,
                detail="입력 문장은 공백일 수 없습니다."
            )
  ```

### 3-2. Exception Handler 기반 전역 예외 처리

> 애플리케이션 전반에서 발생하는 예외를 가로채어, 정해진 형식의 응답으로 변환하는 전역 처리기 => 에러 처리를 전역에서 통일하기 위해 FastAPI가 제공하는 공식 메커니즘
> 검증 실패 요청을 안전하고 통일된 형태로 처리

*에러를 각 라우터나 컨트롤러에서 개별적으로 처리하지 않고 한 곳에서 일관된 규칙으로 처리함*

- **Exception Handle가 필요한 이유**
  | 번호 | 이유 | 설명 |
  |---|---|---|
  | 1 | 에러 응답 형식 통일 | - 모든 에러 응답을 하나의 규칙으로 관리 |
  | 2 | 비즈니스 로직과 에러 처리 로직 분리 | - 컨트롤러: '무엇이 잘못되었는지'만 판단해 예외 발생<br>- Exception Handler: '그 에러를 어떤 상태 코드와 어떤 형식으로 클라이언트에 전달할지' 결정 |
  | 3 | 코드 중복 제거 | - 유지보수 쉬워짐<br>- 응답 형식을 변경하거나 로깅 추가할 때도, 한 곳만 수정하면 됨 |

- **동작**
  ```text
    클라이언트 요청
    -> Pydantic 검증
    -> 라우터나 컨트롤러에서 예외 발생
    -> FastAPI는 해당 예외 타입에 등록된 Exception Handler가 있는지 확인
    -> 등록된 Handler가 있으면 해당 Handler 실행
    -> Handler가 반환한 응답이 최종 응답이 됨
    -> 라우터 함수는 더 이상 실행되지 X
  ```
